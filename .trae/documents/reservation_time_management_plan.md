# 预约时间可视化管理优化方案

## 一、现有实现分析

### 1.1 当前架构概览

根据对项目代码的分析，当前预约时间管理采用**纯配置文件驱动**的方式：

| 组件 | 文件路径 | 职责 |
|------|----------|------|
| 配置文件 | `config.json` / `config.template.json` | 存储用户预约配置（时间、日期、房间、座位） |
| 预约核心逻辑 | `utils/reserve.py` | 处理预约提交和时段处理 |
| 预约记录管理 | `utils/reservation_manager.py` | 管理已预约记录的存储和查询 |
| 主入口 | `main.py` | 读取配置并执行预约流程 |

### 1.2 当前配置结构

```json
{
    "reserve": [
        {
            "username": "学号或手机号",
            "password": "密码",
            "time": ["10:30", "14:15"],    // 开始时间和结束时间
            "roomid": "房间ID",
            "seatid": ["座位号"],
            "daysofweek": ["Monday", "Tuesday", "Wednesday", "Thursday", "Friday"]
        }
    ]
}
```

### 1.3 用户痛点分析

**问题描述**："每次修改预约时间需手动修改配置文件并自行设置时间段，操作流程繁琐"

**具体痛点**：
1. **配置不直观**：用户需要直接编辑 JSON 文件，容易出错
2. **时间格式要求严格**：必须符合 `HH:MM` 格式，手动输入易出错
3. **日期选择繁琐**：需要手动输入英文星期名称
4. **缺乏即时反馈**：修改后无法预览效果，需运行后才能验证
5. **批量管理困难**：多用户场景下配置文件冗长

## 二、功能需求分析

### 2.1 业务规则分析

根据用户提供的预约系统规则，核心规则如下：

| 规则类别 | 规则项 | 规则值 |
|----------|--------|--------|
| **预约规则** | 预约开始时间 | 提前15天的22:00 |
| | 单次预约时长 | 4小时 |
| | 座位预留时长 | 20分钟 |
| | 每日预约次数 | 2次 |
| | 每周预约次数 | 不限 |
| | 暂离次数 | 不限 |
| | 暂离时长 | 120分钟 |
| **违规规则** | 未按时签到 | 记为违约 |
| | 暂离超时未返回 | 记为违约 |
| | 每周违约达5次 | 暂停预约功能 |
| **使用流程** | 完整流程 | 预约座位 → 扫码签到 → 使用完毕退座 |

| S07 | 用户希望导出/导入配置 | 中 |

## 三、技术实现方案

### 3.1 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    前端可视化界面层                          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  配置列表展示  │  时间选择器  │  日期选择器          │    │
│  └─────────────────────────────────────────────────────┘    │
└────────────────────────────┬────────────────────────────────┘
                             │ HTTP API
┌────────────────────────────▼────────────────────────────────┐
│                    后端服务层                               │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  ConfigManager  │  TimeValidator  │  API Router    │    │
│  └─────────────────────────────────────────────────────┘    │
└────────────────────────────┬────────────────────────────────┘
                             │ 文件读写
┌────────────────────────────▼────────────────────────────────┐
│                    数据存储层                               │
│              config.json (配置文件)                         │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 界面设计

#### 3.2.1 界面布局

采用**单页应用**设计，包含以下区域：

| 区域 | 功能 | 位置 |
|------|------|------|
| **顶部导航** | 标题、操作按钮（导出/导入） | 顶部 |
| **规则展示区** | 展示预约规则、违规规则、使用流程说明 | 顶部下方 |
| **配置列表** | 展示所有用户配置项，支持编辑/删除操作 | 左侧主区域 |
| **配置详情面板** | 添加/编辑配置的表单 | 右侧/弹窗 |

#### 3.2.2 核心组件设计

**规则展示组件**：
- 预约规则卡片：展示预约开始时间、时长限制、预留时长等
- 违规规则卡片：展示违约条件和处理措施
- 使用流程卡片：展示预约→签到→退座的完整流程
- 采用折叠式设计，节省空间

**时间选择器组件**：
- 开始时间/结束时间双选择器
- 下拉式时间选择（每15分钟一个选项）
- 实时校验：结束时间必须晚于开始时间
- 时长限制校验：自动检测并提示超过4小时的情况

**日期选择器组件**：
- 星期多选框（周一至周日）
- 快捷选择按钮（工作日/周末/全选）

**配置表单**：
| 字段 | 类型 | 验证规则 |
|------|------|----------|
| username | 文本输入 | 必填，非空 |
| password | 密码输入 | 必填，非空 |
| time[0] | 时间选择 | 必填，格式 HH:MM |
| time[1] | 时间选择 | 必填，格式 HH:MM，晚于开始时间，时长≤4小时 |
| roomid | 文本输入 | 必填，数字 |
| seatid | 文本输入/多选 | 必填 |
| daysofweek | 多选框 | 至少选择1项 |

### 3.3 数据存储方案

**保持向后兼容**，配置文件格式不变，仍使用 `config.json`：

```json
{
    "signin": { ... },
    "reserve": [
        {
            "username": "string",
            "password": "string",
            "time": ["HH:MM", "HH:MM"],
            "roomid": "string",
            "seatid": ["string"],
            "daysofweek": ["Monday", "Tuesday", ...]
        }
    ]
}
```

### 3.4 前后端交互逻辑

#### 3.4.1 API 接口设计

| 接口路径 | HTTP方法 | 功能描述 | 参数 | 返回值 |
|----------|----------|----------|------|--------|
| `/api/config` | GET | 获取所有配置 | 无 | `{ "signin": {...}, "reserve": [...] }` |
| `/api/config/reserve` | POST | 添加新配置 | `{ username, password, time, roomid, seatid, daysofweek }` | `{ "success": true, "message": "添加成功" }` |
| `/api/config/reserve/:index` | PUT | 更新指定配置 | 同上 | `{ "success": true, "message": "更新成功" }` |
| `/api/config/reserve/:index` | DELETE | 删除指定配置 | 无 | `{ "success": true, "message": "删除成功" }` |
| `/api/config/export` | GET | 导出配置文件 | 无 | JSON 文件流 |
| `/api/config/validate` | POST | 校验配置格式 | 配置对象 | `{ "valid": true, "errors": [] }` |

#### 3.4.2 数据流

```
用户操作 → 前端界面 → API请求 → 后端处理 → 配置文件
                                    ↓
                               返回结果 → 前端展示反馈
```

### 3.5 技术栈选择

| 层级 | 技术 | 版本 | 选择理由 |
|------|------|------|----------|
| 前端框架 | Vue.js | 3.x | 轻量、响应式、学习曲线平缓 |
| UI组件库 | Element Plus | 2.x | 组件丰富、文档完善、与Vue3无缝集成 |
| 后端框架 | Flask | 2.x | 轻量、易用、适合快速开发，跨平台支持好 |
| 语言 | Python | 3.8+ | 与现有项目保持一致，跨平台特性优秀 |
| 构建工具 | Vite | 5.x | 快速构建、热更新 |
| 跨平台打包 | PyInstaller | 6.x | 支持打包为Windows/Linux可执行文件 |
| Android包装 | Nativefier/Tauri | - | 将Web应用打包为Android应用 |

### 3.6 跨平台适配方案

#### 3.6.1 平台支持策略

| 平台 | 部署方式 | 特点 |
|------|----------|------|
| **Windows** | 打包为exe可执行文件 | 双击运行，内置Web服务器和前端资源 |
| **Linux** | 打包为AppImage/deb/rpm | 支持主流发行版，无需安装依赖 |
| **Android** | WebView封装或Tauri打包 | 移动设备访问，支持触控操作 |
| **通用Web** | 直接部署为Web服务 | 跨平台浏览器访问 |

#### 3.6.2 打包方案

**Windows打包**：
```bash
pyinstaller --onefile --windowed --add-data "frontend/dist;frontend/dist" app.py
```

**Linux打包**：
```bash
pyinstaller --onefile --windowed --add-data "frontend/dist:frontend/dist" app.py
```

**Android打包**（使用Nativefier）：
```bash
nativefier "http://localhost:5000" --platform android --name "预约管理"
```

#### 3.6.3 界面适配

- **响应式设计**：使用Element Plus响应式布局，适配不同屏幕尺寸
- **触控优化**：按钮尺寸≥48px，便于移动端触控
- **字体适配**：使用相对单位，确保在不同DPI下清晰可读
